# 📋 [최종 기획서] To-Do RPG

**프로젝트명:** To-Do RPG (일상 관리 CLI RPG 시스템)
**제출자:** 컴퓨터소프트웨어공학과 3학년 나민주
**개발 환경:** VS Code / Ubuntu Linux 환경
**사용 언어:** Python 3 (json, os, time 내장 라이브러리 활용)


## 1. 목적 (Purpose)

* **문제 정의:** 기존의 투두리스트(To-Do List) 프로그램들은 단순 텍스트 체크 방식에 그쳐 사용자가 흥미를 쉽게 잃고 일과를 미루는 경향이 있음.
* **해당 과제의 목적:** 일상의 할 일(과제, 운동, 공부 등)을 게임 속 '퀘스트(Quest)'로 치환하고, 성취에 따른 보상(XP, 골드)과 실패에 따른 페널티(HP 감소)를 부여해 일상 관리 행위 자체를 하나의 몰입감 있는 RPG 게임처럼 즐기도록 유도함.


## 2. 주요 기능 및 CRUD 설계

* **Create (등록):** 오늘 수행할 일일 퀘스트(할 일) 등록 및 중요도/난이도(상/중/하) 설정하여 객체 생성.
* **Read (조회):** 터미널 상단에 캐릭터 스탯 상태 창(Lv, XP, HP, Gold) 및 특수문자(`■`, `□`) 기반의 실시간 프로그레스 바를 출력하고, 하단에 진행 중인 퀘스트 목록을 정렬하여 대시보드 형태로 조회.
* **Update (수정):** 퀘스트 완료 체크 시 난이도별 경험치와 골드를 정산하고 레벨업 판정 알고리즘 구동. 할 일 문구의 인라인 수정 기능 포함.
* **Delete (삭제):** 일정이 취소되거나 잘못 등록된 일일 퀘스트를 예외 처리 절차를 거쳐 안전하게 삭제.
* **System (상점):** 미완료 퀘스트 발견 시 HP가 깎이는 패널티 알고리즘 및 모은 골드로 자신을 위한 셀프 보상을 구매하는 미니 상점 시스템.


## 3. 소프트웨어 아키텍처 및 품질 지표 (Architecture & Code Quality)

본 프로젝트는 확장성과 테스트 용이성을 극대화하기 위해 다음과 같은 객체지향 설계 지표 및 클린 아키텍처 규칙을 엄격히 준수한다.

### 🧱 아키텍처 설계 규칙

1. **단일 책임 원칙 (SRP):** 데이터 구조(Models), 데이터 영구 보존(Repositories), 핵심 게임 비즈니스 로직(Services), 터미널 화면 제어(Main UI) 레이어를 명확히 분리한다.
2. **리포지토리 패턴 및 성능 최적화:** 로컬 JSON 파일 입출력 시 딕셔너리(`dict`) 키 인덱싱을 활용하여 데이터 검색 및 수정 성능을 $O(1)$로 보장한다. (무분별한 리스트 순회 $O(n)$ 지양)
3. **의존성 주입 (Dependency Injection):** `RpgService` 및 `ShopService` 생성 시 리포지토리 객체를 생성자 인자로 주입받는 구조로 설계하여 향후 Mock 객체를 통한 단위 테스트가 가능하도록 조립한다.
4. **견고한 에러 처리:** 단순 `print` 방식의 에러 노출을 금지하고, 비즈니스 규칙 위반 시 커스텀 예외 클래스(`src/exceptions/`)를 발생시켜 메인 루프에서 유효성 검사(`try-except`)로 안전하게 낚아챈다.

### 📐 코드 품질 지표 (Metrics)

* **순환 복잡도 (Cyclomatic Complexity):** 메서드당 10 이하 유지
* **함수 길이:** 모든 함수 및 메서드는 30줄 이하로 제한
* **네이밍 컨벤션:** * 약어 사용 금지 (예: `uid` ➡️ `user_id`, `q` ➡️ `quest`)
* 함수명은 동사로 시작 (`create_quest`, `validate_input`)
* Boolean 리턴 함수는 접두사 필수 (`is_levelup_condition`, `can_afford_item`)
* 상수는 대문자 스네이크 표기법 적용 (`MAX_HP = 100`)


## 4. 구조화된 파일 레이아웃 (Directory Layout)

```text
todo_rpg/
├── src/
│   ├── models/          # 도메인 모델 데이터 구조 (User.py, Quest.py)
│   ├── repositories/    # 파일 스트림 입출력 엔진 (UserRepository.py, QuestRepository.py)
│   ├── services/        # 비즈니스 규칙 및 검증 (RpgService.py, ShopService.py, Validator.py)
│   └── exceptions/      # 시스템 커스텀 예외 정의 (GameExceptions.py)
├── tests/               # 단위 테스트 스크립트 팩
├── main.py              # 프로그램 진입점 및 터미널 UI 메인 루프 
├── README.md            # 기술 문서 및 가이드라인
└── GEMINI.md            # 제미나이 CLI 프롬프트 컨텍스트 파일

```


## 5. 개발 일정 및 시간 계획 (Timeline)

* **1일차: 캐릭터 도메인 모델 및 커스텀 예외 설계 (2시간)**
* `User`, `Quest` 데이터 모델 정립 및 `InsufficientGoldError`, `HeroDeadError` 같은 예외 클래스 빌드.

* **2일차: O(1) 성능의 리포지토리 레이어 구축 (3시간)**
* `todo_rpg.json` 로컬 파일과 연동되는 파일 입출력 스트림 구현 및 딕셔너리 기반의 초고속 검색/저장 엔진 코딩.

* **3일차: 게임 비즈니스 로직 및 유효성 검증 서비스 연동 (3시간)**
* 난이도별 보상 정산 및 레벨업 판정 알고리즘, 공백 입력 차단 `Validator`, 골드 소모형 미니 상점 구현.

* **4일차: UI 대시보드 조립 및 최종 디버깅 (1.5시간)**
* `os.system("clear")` 기반 화면 제어, 특수문자(`■`, `□`) 활용 상태 바 마크업 및 종합 예외 처리 적용.
