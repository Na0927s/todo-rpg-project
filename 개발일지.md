# 1번째 (1일차)
  1. 🎯 프로젝트 개요 및 목표
   * 목적: 지루할 수 있는 일상적인 투두리스트를 RPG 게임의
     '퀘스트'로 치환하여, 완료 시 보상(XP, 골드)과 미완료 시
     패널티(HP 감소)를 통해 몰입감 있는 일정 관리를 돕는 CLI
     프로그램 개발.
   * 아키텍처: 단일 책임 원칙(SRP)과 클린 아키텍처를 기반으로
     Model, Repository, Service, UI 계층을 엄격히 분리하여
     확장성 및 테스트 용이성을 극대화함.

  2. 🛠️ 진행 상황 및 구현 내용
  현재 계획된 1번째 개발 목표인 "캐릭터 도메인 모델 및 커스텀
  예외 설계"가 성공적으로 완료되었습니다.

   * 도메인 모델 (Models) 구현 완료
     * User 모델 (src/models/user.py): 캐릭터의 상태를 관리하기
       위해 Level, XP, HP, Gold 속성을 정의했습니다. 상태
       조회를 위한 is_dead() 메서드와 스탯 증감을 처리하는
       decrease_hp(), add_xp(), spend_gold(), add_gold()
       메서드를 구현하여 캡슐화를 충족했습니다.
     * Quest 모델 (src/models/quest.py): 할 일 관리를 위한 ID,
       내용, 난이도(상/중/하), 완료 여부를 저장하는 구조를
       설계했습니다. 상태 변경을 위한 complete_quest()와
       Boolean 반환형 네이밍 규칙을 준수한 is_finished()
       메서드를 포함했습니다.
   * 커스텀 예외 처리 (Exceptions) 구축 완료
     * GameExceptions.py (src/exceptions/GameExceptions.py):
       비즈니스 로직 예외 상황을 안전하게 제어하기 위해 2가지
       커스텀 예외를 작성했습니다.
       * InsufficientGoldError: 상점 이용 시 보유 골드가 부족할
         경우 발생. (User.spend_gold 내부 로직에 연동 적용
         완료)
       * HeroDeadError: HP가 0 이하로 떨어져 캐릭터가 사망할 때
         발생. (User.decrease_hp 내부 로직에 연동 적용 완료)

  3. 🔍 코드 품질 및 컨벤션 검토
   * 사전에 정의된 is_, has_ 등의 Boolean 메서드 네이밍 규칙이
     잘 적용되었습니다.
   * 예외를 단순히 print로 처리하지 않고 도메인 내부에서
     명시적인 raise Exception 구조를 사용하여, 향후 Service /
     UI 레이어의 try-except 블록에서 제어할 수 있도록 견고하게
     작성되었습니다.

  4. 🚀 향후 개발 계획 (Next Steps)
   * **파일 시스템 및 리포지토리 레이어 구축:** 
     * todo_rpg.json 파일을 데이터베이스로 사용하는 입출력
       스트림 구현.
     * 리스트 순회($O(n)$) 대신 딕셔너리(dict) 키 인덱싱을
       활용하여 $O(1)$ 성능의 데이터 검색 및 저장 엔진 개발
       (UserRepository, QuestRepository).
   * **비즈니스 로직 연동:** RpgService를 통해
     난이도별 보상 정산 로직, 레벨업 알고리즘, 상점 시스템
     구현.
   * **메인 UI 구성:** os.system("clear") 및
     특수문자를 이용한 대시보드 상태창 렌더링 및 main.py 진입점

## 2. 주요 트러블슈팅 및 해결 과정

### ① VS Code 파이썬 확장 프로그램(Extension) 연동 오류
* **문제 상황:** 로컬(Windows)에는 파이썬 확장이 설치되어 있었으나, 실제 작업 공간인 WSL 우분투 환경과 연동되지 않아 노란색 경고 문구 발생. 하단 바에 표시된 윈도우용 파이썬 버전과의 혼선이 있었음.
* **원인 분석:** VS Code 원격 개발 시, 소스 코드가 실행되는 환경(WSL) 내부에도 개별적으로 확장 프로그램이 설치되어야 정상 작동함.
* **해결 방법:** 
  1. WSL 터미널에서 필수 패키지 관리 도구 설치 (`sudo apt install python3-pip python3-venv -y`)
  2. VS Code 확장 탭에서 `[WSL: Ubuntu-24.04에 설치]` 파란색 버튼을 클릭하여 우분투 환경 내부에 직접 확장 컴포넌트 이식 완료.

### ② 리눅스 파이썬 명령어 인식 문제 (`Command 'python' not found`)
* **문제 상황:** 터미널에서 `python app.py` 입력 시 명령어를 찾을 수 없다는 에러 발생.
* **원인 분석:** 우분투 환경에서는 기본적으로 `python`이 아닌 `python3` 명칭을 명확히 명시해 주어야 인지함.
* **해결 방법:** `python3 app.py` 명령어로 대체하여 정상적으로 파이썬 스크립트 실행 시작.

### ③ 외부 라이브러리 누락 에러 (`ModuleNotFoundError: No module named 'flask'`)
* **문제 상황:** Flask 패키지가 없어 서버 실행이 차단됨.
* **원인 분석:** 우분투 시스템 자체에 Flask 라이브러리가 깔려있지 않음.
* **해결 방법:** 독립된 가상환경(`venv`) 방을 생성 및 활성화한 후 `pip install flask`를 통해 필요한 종속성을 안전하게 격리 설치함.

### ④ Git 커밋 파일 폭탄 및 최적화 (`1K+` 변경 사항 발생)
* **문제 상황:** 소스 제어 탭에 1,000개가 넘는 커밋 대기 파일이 생성되어 관리가 불가능한 상태 발생.
* **원인 분석:** 파이썬 외부 패키지가 담긴 가상환경 폴더(`venv/`)와 자동 생성되는 캐시 폴더(`__pycache__/`) 안의 수천 개 시스템 파일이 Git 추적 대상에 포함됨.
* **해결 방법:** 프로젝트 최상위 루트 폴더에 `.gitignore` 파일을 생성하고 아래 예외 목록을 등록하여 코드 파일들만 추적하도록 최적화함.
